5 # Presentation # {#pres_main}
6 
7 Presentation refers to the part of the application that is visible to the user -
8 the _user interface_ - and the handling of user interaction.
9 
10 ## Application ## {#pres_application}
11 
12 Web applications typically implement a _request-response pattern_, where a
13 client sends a
14 ([HTTP](http://en.wikipedia.org/wiki/Hypertext_Transfer_Protocol)-)request to the
15 application, which returns a response after processing it. wCMF encapsulates the
16 described procedure inside the
17 \link wcmf::lib::presentation::Application `Application`\endlink class. The
18 following code demonstrates the usage of this class in the main entry script of
19 wCMF's default application.
20 
21 ~~~~~~~~~~~~~{.php}
22 $application = new Application();
23 try {
24  // initialize the application
25  $request = $application->initialize(WCMF_BASE.'app/config/', 'config.ini', '', '', 'cms');
26 
27  // run the application
28  $response = $application->run($request);
29 }
30 catch (Exception $ex) {
31  try {
32  $application->handleException($ex, isset($request) ? $request : null);
33  }
34  catch (Exception $unhandledEx) {
35  echo("An unhandled exception occured. Please see log file for details.");
36  }
37 }
38 ~~~~~~~~~~~~~
39 
40 The example shows the three important public methods of the
41 \link wcmf::lib::presentation::Application `Application`\endlink class:
42 
43 - The \link wcmf::lib::presentation::Application::initialize `initialize`\endlink
44  method is used to __setup__ the
45  \link wcmf::lib::presentation::Application `Application`\endlink. It returns a
46  \link wcmf::lib::presentation::Request `Request`\endlink instance, that may be
47  modified before execution.
48 - The \link wcmf::lib::presentation::Application::run `run`\endlink method is
49  called to __execute__ the given request. The method returns a
50  \link wcmf::lib::presentation::Response `Response`\endlink instance, that is not
51  used in this example.
52 - The \link wcmf::lib::presentation::Application::handleException `handleException`\endlink
53  method is called, if an __exception__ occurs. The method rolls back the database
54  transaction and calls the `failure` action.
55 
56 The details of request execution are the topic of the next section.
57 
58 ## Request processing ## {#pres_request}
59 
60 The \link wcmf::lib::presentation::Request `Request`\endlink instance created
61 on initialization of the application provides all information about the incoming
62 HTTP request, that is necessary for execution. Upon execution, the following
63 actions are performed:
64 
65 1. The \link wcmf::lib::presentation::Request `Request`\endlink instance is passed
66  to \link wcmf::lib::presentation::ActionMapper `ActionMapper`\endlink for further
67  processing.
68 2. The \ref arch_actionkey is determined from the request parameters.
69 3. \link wcmf::lib::security::PermissionManager `PermissionManager`\endlink is
70  asked to authorize the action key for the current user (see \ref perm_check).
71 4. If authorization is successful, the request data is transformed into the internal
72  application format (see \ref pres_format).
73 5. The \link wcmf::lib::presentation::Controller `Controller`\endlink instance
74  matching the request is determined (see \ref pres_routing) and executed.
75 6. The \link wcmf::lib::presentation::Response `Response`\endlink instance is
76  obtained after execution.
77 7. The response data is transformed into the requested response format (see
78  \ref pres_format).
79 8. Execution returns to step 3, if a valid action key is contained in the response
80  data and terminates, if not.
81 
82 ### Formats ### {#pres_format}
83 
84 wCMF is designed to be able to consume various request formats and produce
85 several response formats. While some clients communicate using
86 [JSON](http://en.wikipedia.org/wiki/JSON) format, others might prefer to encode
87 data in [XML](http://en.wikipedia.org/wiki/XML).
88 The \link wcmf::lib::presentation::format::Formatter `Formatter`\endlink class
89 is used to determine the required format and delegate the actual formatting to
90 the correct \link wcmf::lib::presentation::format::Format `Format`\endlink
91 implementation. wCMF currently provides the following __implementations__:
92 
93 - \link wcmf::lib::presentation::format::impl::HtmlFormat `HtmlFormat`\endlink
94  expects all data to be sent as key-value-pairs. Object data are transferred in
95  parameters named `value-`<em>name-oid</em> (e.g. `value-title-Book:3`).
96  Responses in this format are rendered as HTML views (see \ref pres_views).
97 - \link wcmf::lib::presentation::format::impl::JsonFormat `JsonFormat`\endlink
98  handles JSON encoded request data and encodes response data into the same format.
99 - \link wcmf::lib::presentation::format::impl::SoapFormat `SoapFormat`\endlink
100  is used together with the [NuSOAP](http://sourceforge.net/projects/nusoap/)
101  library to implement a SOAP interface.
102 - \link wcmf::lib::presentation::format::impl::NullFormat `NullFormat`\endlink
103  is used internally, if no formatting is required, e.g. if one controller calls
104  another controller.
105 
106 If not explicitely set, the request and response format is automatically
107 determined from the __HTTP headers__ sent with the request:
108 
109 - The `Content-Type` header defines the __request__ format
110 - The `Accept` header defines the __response__ format
111 
112 Formats are defined in the `Formats` configuration section as shown in the following
113 example:
114 
115 ~~~~~~~~~~~~~{.ini}
116 [Formats]
117 html = $htmlFormat
118 null = $nullFormat
119 
120 [HtmlFormat]
121 __class = wcmf\lib\presentation\format\impl\HtmlFormat
122 
123 [NullFormat]
124 __class = wcmf\lib\presentation\format\impl\NullFormat
125 ~~~~~~~~~~~~~

127 ### Routing ### {#pres_routing}
128 
129 Routing is the process of selecting the correct
130 \link wcmf::lib::presentation::Controller `Controller`\endlink for a given request.
131 wCMF distinguishes between _internal_ and _external_ routing.
132 
133 #### Internal routing #### {#pres_routingint}
134 
135 Internal routing takes place after the \link wcmf::lib::presentation::Request `Request`\endlink
136 instance is created and initialized. wCMF inspects the **action key** formed from
137 it's _sender_, _context_ and _action_ parameters (see \ref arch_actionkey) to
138 determine the controller to be executed next. The mapping of action keys to
139 controllers is defined in the `ActionMapping` configuration section.
140 
141 If the executed controller together with the _context_ and _action_ parameters of
142 the response match another action key, the corresponding controller
143 will be executed afterwards. This allows to chain several actions together.
144 
145 #### External routing #### {#pres_routingext}
146 
147 The mapping of the current **request uri** to an action key is called external
148 routing. The default logic is implemented in the
149 \link wcmf::lib::presentation::impl::DefaultRequest::initialize `DefaultRequest::initialize`\endlink
150 method. The method matches the _path part_ of the request uri against the entries
151 of the `Routes` configuration section. Variables declared in path segments will
152 be automatically passed as request parameters.
153 
154 The following example configuration taken from the default application illustrates
155 the concept:
156 
157 ~~~~~~~~~~~~~{.ini}
158 [Routes]
159 / = action=cms
160 /rest/{language}/{className} = action=restAction&collection=1
161 /rest/{language}/{className}/{id|[0-9]+} = action=restAction&collection=0
162 ~~~~~~~~~~~~~
163 
164 - The first entry is matched by the root path, which is then mapped to the _cms_
165 action - corresponding to the action key <em>??cms</em>.
166 - The second entry defines the _language_ and _className_ variables (surrounded
167 by curly braces) and would be matched by the request uris <em>/rest/de/Author</em>
168 or <em>/rest/en/Book</em>. The executed action would be _restAction_.
169 - The _id_ variable in the third entry must be an integer because of the regular
170 expression constraint `[0-9]+`.
