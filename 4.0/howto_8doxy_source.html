7 \subsection howtotype Definition of the data model
8 
9 The data model defines the domain classes. In principle the following steps are necessary
10 to to make the domain classes available in the application:
11 
12 -# Definition and creation of the database tables.@n
13 -# Implementation of the PersistenceMapper classes, which map the domain classes to the tables.@n
14 -# Creation of special domain classes, if the Node class is not sufficient.@n
15 
16 Basically it doesn't matter, in which format the application data is stored. To load and store
17 data application developers solely have to communicate with the PersistenceFacade.
18 The PersistenceMapper classes have the actual knowledge of how to access the data.
19 This abstraction makes it for instance possible to easily migrate the data storage from file based
20 to databased. Only the mappers have to be exchanged. On the other hand it's also possible - an
21 adequate implementation of the mappers assumed - to access arbitrary database schemes, which
22 eases the connection of the wCMF to existing data sources.
23 
24 In practice it has been proved as advantageous to use a database as storage medium, because it has
25 a better performance and can handle bigger amounts of data much better than XML files can.@n
26 It is recommended to define one table for each domain class in the database scheme (e.g. @em author, @em article).
27 On one hand it makes the import and export from and to other applications easier and on the other
28 hand the access to the data is optimized compared to storing all data in one table, because then
29 additional joins would be required to determine data types and other meta information. In addition the
30 table columns can be more easily adjusted to the needs of the domain classes (in an extreme case all data
31 must be stored as BLOB in an universal data table). NodeToSingleTableMapper implements the storage of all
32 data in one table, which - for the said reasons - isn't recommended.
33 
34 Proceeding on these assumptions the creation of the data model becomes a lot simpler.
35 
36 \subsubsection howtotables Definition of the database tables
37 
38 We start with step 1. Assuming we're in the fortunate situation of defining the database tables
39 ourselves. Then we proceed as follows.
40 
41 - For each domain class we create one table.
42 - For each attribute of a domain class we define a row with the appropriate datatype.
43 
44 Further attributes follow a scheme, which is predefined by the NodeUnifiedRDBMapper. This
45 makes the creation of the mappers easier:
46 
47 - Each table gets a primary key named @em id (int(11))
48 - If one table depends on another, it gets a foreign key @em fk_table_id (int(11)), where
49  @em table stands for the name of the parent table.
50 
51 If an N to M relation should be established between two domain classes (tables), a connection
52 table must be defined. This table contains two primary keys, which point to the two connected
53 tables.
54 
55 As an example serves a domain class @em Article, which is related to an @em Author:
56 @verbatim
57  CREATE TABLE `Article ` (
58  `id` int(11) NOT NULL default '0',
59  `fk_author_id` int(11) default NULL,
60  `headline` VARCHAR(255),
61  `text` TEXT,
62  `sortkey` INT(3),
63  PRIMARY KEY (`id`)
64  ) TYPE=MyISAM;
65 @endverbatim
66 
67 If the database tables already exist, this step is skipped. This however means, that more work
68 must be put into the implementation of the PersistenceMapper.
69 
70 \subsubsection howtomapper Implementation of the PersistenceMapper
71 
72 Of course it's always possible to write own mappers. These must inherit from PersistenceMapper
73 and implement the abstract methods given there.
74 Since the framework already provides mappers, it's easier to use these as base classes. In case of a relational
75 database it's possible to derive custom mappers from NodeRDBMapper. Then the methods for defining
76 the SQL statements must be implemented.
77 
78 Assuming the database tables are created by the scheme above, we can proceed to the next step
79 and use NodeUnifiedRDBMapper as baseclass for our mappers. After that we have to implement the
80 methods RDBMapper::getType, NodeRDBMapper::createObject, NodeUnifiedRDBMapper::getTableName,
81 PersistenceMapper::getPkNames, NodeUnifiedRDBMapper::getMyFKColumnNameImpl and
82 NodeUnifiedRDBMapper::getObjectDefinitionImpl. These methods merely ask for properties of
83 the domain class. The example shows the mapper for the table @em Article mentioned above:
84 @verbatim
85  class ArticleRDBMapper extends NodeUnifiedRDBMapper
86  {
87  /**
88  * @see RDBMapper::getType()
89  */
90  function getType()
91  {
92  return 'Article';
93  }
94  /**
95  * @see NodeRDBMapper::createObject()
96  */
97  function &createObject($oid=null)
98  {
99  return new Article($oid);
100  }
101  /**
102  * @see NodeUnifiedRDBMapper::getTableName()
103  */
104  function getTableName()
105  {
106  return 'Article';
107  }
108  /**
109  * @see PersistenceMapper::getPkNames()
110  */
111  function getPkNames()
112  {
113  return array('id' => DATATYPE_IGNORE);
114  }
115  /**
116  * @see NodeUnifiedRDBMapper::getMyFKColumnNameImpl()
117  */
118  function getMyFKColumnNameImpl($parentType)
119  {
120  // start from the most specific
121  if ($this->getType() == 'Article' && $parentType == 'Author') return 'fk_author_id';
122  if ($parentType == 'Author') return 'fk_author_id';
123  return '';
124  }
125  /**
126  * @see NodeUnifiedRDBMapper::getOrderBy()
127  */
128  function getOrderBy()
129  {
130  return array();
131  }
132  /**
133  * @see NodeUnifiedRDBMapper::getObjectDefinitionImpl()
134  */
135  function getObjectDefinitionImpl()
136  {
137  $nodeDef = array();
138  $nodeDef['_properties'] = array
139  (
140  array('name' => 'is_searchable', 'value' => false),
141  );
142  $nodeDef['_datadef'] = array
143  (
144  /*
145  * Value description:
146  */
147  array('name' => 'id', 'app_data_type' => DATATYPE_IGNORE, 'column_name' => 'id',
148  'db_data_type' => 'INT(11) NOT NULL', 'default' => '', 'restrictions_match' => '', 'restrictions_not_match' => '',
149  'restrictions_description' => '', 'is_editable' => false, 'input_type' => 'text', 'display_type' => 'text'),
150  /*
151  * Value description:
152  */
153  array('name' => 'fk_author_id', 'app_data_type' => DATATYPE_IGNORE, 'column_name' => 'fk_author_id',
154  'db_data_type' => 'INT(11)', 'default' => '', 'restrictions_match' => '', 'restrictions_not_match' => '',
155  'restrictions_description' => '', 'is_editable' => false, 'input_type' => 'text', 'display_type' => 'text'),
156  /*
157  * Value description:
158  */
159  array('name' => 'headline', 'app_data_type' => DATATYPE_ATTRIBUTE, 'column_name' => 'headline',
160  'db_data_type' => 'VARCHAR(255)', 'default' => '', 'restrictions_match' => '', 'restrictions_not_match' => '',
161  'restrictions_description' => '', 'is_editable' => false, 'input_type' => 'text', 'display_type' => 'text'),
162 
163  ...
164  );
165  $nodeDef['_ref'] = array
166  (
167  /*
168  * Value description:
169  */
170  array('name' => 'author_name', 'ref_type' => 'Author', 'ref_value' => 'name', 'ref_table' => 'Author',
171  'id_column' => 'id', 'fk_columns' => 'fk_article_id', 'ref_column' => 'name')
172  );
173  $nodeDef['_parents'] = array
174  (
175  array('type' => 'Author', 'is_navigable' => false, 'table_name' => 'Author', 'pk_columns' => array('id'),
176  'fk_columns' => 'fk_author_id')
177  );
178  $nodeDef['_children'] = array
179  (
180  array('type' => 'Image', 'minOccurs' => 0, 'maxOccurs' => 'unbounded', 'aggregation' => false, 'composition' => true,
181  'is_navigable' => false, 'table_name' => 'Image', 'pk_columns' => array('id'), 'fk_columns' => 'fk_article_id',
182  'order_by' => array())
183  );
184  return $nodeDef;
185  }
186  }
187 @endverbatim
188 
189 The method NodeUnifiedRDBMapper::getObjectDefinitionImpl is the most important method. It supplies an
190 associative array, in which the attributes of the domain class (@em _datadef) the parent and the child
191 domain classes (@em _children) are defined. In addition the Article contains a reference to
192 the name of the Author (@em _ref), which allows to access the Author without loading it.@n
193 Detailed information on the implemetation can be found under NodeUnifiedRDBMapper::getObjectDefinitionImpl.

197 \subsubsection howtodomainclass Creating the domain classes
198 
199 The framework's PersistenceMapper work with the class Node, a subclass of PersistentObject.
200 Different domain classes are created by specifying the type attribute (Node::getType) and
201 a set of attributes (Node::getValueNames).
202 So the class Node is a generic data container, into which data can be put by the method
203 Node::setValue and retrieved by the method Node::getValue. For most applications this will
204 do.
205 
206 If however more specialized domain classes are required for the application, they must
207 be created by the PersistenceMapper. In this case subclasses of NodeRDBMapper must
208 override the method NodeRDBMapper::createObject, which in turn creates instances of the
209 special domain class.

278 \subsection howtoviews Programming the views
279 
280 Views are implemented as HTML pages (defined in the view templates), which typically contain a form,
281 which displays the data to be modified. For programming dynamic parts and to access
282 application data the <a href="http://smarty.php.net/" target="_blank">Smarty</a> template
283 language is used.@n
284 By default the views are stored as .tpl files in the directory /application/inlcude/views
285 (see @ref secsmarty). In the directory /wcmf/application/views those views are stored, which
286 the framework uses for its standard application. These are the basis for the programming of
287 custom views.@n
288 
289 In the view templates all data, which was passed to the view instance is accessible (see
290 @ref howtocontroller). In the simplest case these can be displayed via @em {$variable}. In
291 addition object data can be accessed by using @em {$object->getValue(...)}. By setting
292 @em debugView = 1 (see @ref seccms) in the configuration file Smarty will display the data,
293 which is available in the template, in an external window.@n
294 
295 The data displayed in the view's form is available to the following controller. Some
296 (hidden) input fields should always exist. They are defined in the file
297 /wcmf/application/views/formheader.tpl, which - to simplify matters - should be reused.@n
298 For handling the form data some JavaScript functions are provided (and documented) in
299 the file /wcmf/blank/script/common.js.
300 
301 In the directory /wcmf/lib/presentation/smarty_plugins the framework defines extensions
302 of the Smarty template language:
303 
304 - Function @em translate for localization of strings@n
305  e.g.: {translate text="Logged in as %1% since %2%" r0="$login" r1="$logindate"}
306 - Function @em sessionvalue to get a session variable@n
307  e.g.: {sessionvalue name="platform"}
308 - Resource @em lib to integrate templates with a relative path to the framework's
309  root directory (/wcmf)@n
310  e.g.: {include file="lib:application/views/formheader.tpl"}@n
311  ...

313 \subsection howtocontroller Programming the controllers
314 
315 Controllers execute the user-defined actions. In order to implement custom controllers
316 a class must be derived from the baseclass Controller, which implements the methods
317 Controller::hasView and Controller::executeKernel.
318 
319 The Request instance passed to the Controller::initialize method provides all data of the
320 preceeding view's input fields to the controller. The names of the input fields are the names
321 of the request values. The controller in turn can pass data to the view by setting them on
322 the Response instance.
323 
324 The method Controller::hasView returns @em true or @em false, whether a view is displayed
325 or not (the return value can differ depending on the context or action, for an example see LoginController).@n
326 The method Controller::executeKernel executes the actual action. In this method application
327 data is loaded, modified, created, deleted and where required passed to the view for display
328 or to the next controller to proceed.
329 The method either returns @em false, which means, that the ActionMapper should call no further
330 controller or true. In the latter case the ActionMapper determines the next controller
331 from the context and action values of the response (see @ref actionkey).
332 This means if a view should be displayed, the method must return @em false.
333 
334 While programming custom controllers often the methods Controller::initialize and
335 Controller::validate are overridden in order to carry out initializations or to
336 validate provided data.
337 
338 The framework's controllers are located in the directory /wcmf/application/controller.

340 \subsection howtoapplication Programming the application
341 
342 A web application typically consists of several input masks (views), which
343 are used to create, modify and delete data. The application is defined by the actions
344 executable in the individual input masks. Thereby the framework makes no difference
345 between actions used for data handling and those used to navigate or e.g. initiate the
346 export of data.
347 
348 The definition of an action requires the following steps:
349 
350 -# @ref howtoapplication1
351 -# @ref howtoapplication2
352 -# @ref howtoapplication3
353 -# @ref howtoapplication4
354 -# @ref howtoapplication5
355 
356 As an example we use the action for displaying an @em article node in order to
357 edit it. Let's look at the individual steps:
358 
359 \subsubsection howtoapplication1 Definition of the action name
360 We name the action @em editArticle. This name need not to be unique in the whole
361 application. The ActionMapper only requires the name (and the @ref actionkey defined
362 by the action) to find the next appropriate controller.
363 
364 \subsubsection howtoapplication2 Creating the button to trigger the action
365 In order to display the data the application must know which article is selected.
366 This is exactly defined by it's @ref oid. The data transfer between the input
367 masks is achieved by the HTTP POST mechanism, i.e. a (hidden) input field must exist,
368 which contains the oid of the article to be displayed. Since for most applications it's often
369 necessary to transfer an oid, the framework defines a standard field @em oid in each
370 view (see file /wcmf/application/views/formheader.tpl), which can easily be set
371 by the JavaScript function @em doDisplay (/wcmf/blank/script/common.js).
372 
373 The action is triggered upon submission of the input form. Another JavaScript function
374 (@em submitAction) simplifies the execution. The form data is passed to the main.php
375 script, which delegates the further execution to the ActionMapper. The link to execute
376 the action could look like this:
377 @verbatim <a href="javascript:setContext('article'); doDisplay('{$article->getOID()}');
378  submitAction('editArticle');">{translate text="edit"}</a> @endverbatim
379 For details on programming the views see @ref howtoviews.
380 
381 \subsubsection howtoapplication3 Customizing the configuration file
382 To determine the controller, which carries out the action, the ActionMapper requires
383 an appropriate entry in the configuration file (see @ref secactionmapping). If
384 the controllers name is @em ArticleController, the entry could look like this:
385 @verbatim
386  [actionmapping]
387  ??editArticle = ArticleController
388 @endverbatim
389 Don't forget to introduce the ArticleController in the configuration section @ref secclassmapping.
390 
391 Additionally the ArticleController should display a view for editing the article. If we
392 name this view @em article.tpl, the configuration entry would look like the following (see
393 @ref secviews):
394 @verbatim
395  [views]
396  ArticleController?? = article.tpl
397 @endverbatim
398 
399 \subsubsection howtoapplication4 Implementing the action as a controller
400 The action is executed in the controller - in this example in the @em ArticleController class.
401 Since the controller should display a view with the article's data, we first must specify that
402 the controller has a view and second the data of the article must be passed to the view.@n
403 At first however it must be assured, that the controller receives an oid. This happens in
404 the method Controller::validate, which searches for the entry in the passed data:
405 @verbatim
406  function validate()
407  {
408  if ($this->_request->getValue('oid') == '')
409  {
410  $this->setErrorMsg("No 'oid' given in data.");
411  return false;
412  }
413  return true;
414  }
415 @endverbatim
416 
417 We declare the existence of a view in the method Controller::hasView:
418 @verbatim
419  function hasView()
420  {
421  return true;
422  }
423 @endverbatim
424 
425 Finally the action is executed in the method @em Controller::executeKernel. Here the
426 controller loads the data and provides it to the view for display by setting it on the
427 response instance:
428 @verbatim
429  function executeKernel()
430  {
431  $persistenceFacade = &PersistenceFacade::getInstance();
432 
433  // load model
434  $article = &$persistenceFacade->load($this->_request->getValue('oid'), BUILDDEPTH_INFINITE);
435 
436  // assign model to view
437  $this->_response->setValue('article', $article);
438 
439  // stop processing chain
440  return false;
441  }
442 @endverbatim
443 
444 It's important that the method returns false, since this causes the ActionMapper
445 to end the execution and wait for user input. The display of the view is done by
446 the framework.
447 
448 \subsubsection howtoapplication5 Displaying data in the view
449 After the controller has provided the view with the data, the view can display the data.
450 In our case after the ArticleController has been executed a variable @em article is
451 known to the view, which matches the article node.@n
452 The programming of the views is done in HTML together with the Smarty template language.
453 The file @em article.tpl could contain the following line:
454 @verbatim article name: {$nodeUtil->getInputControl($article, "name")} @endverbatim
455 
456 In the curly brackets you can find Smarty code, which calls the method NodeUtil::getInputControl.
457 This method displays the input control (in our case a textfield), which corresponds to the
458 article's @em name attribute, in the HTML page. In the same manner the other attributes can be handled.

460 \subsection howtorights Rights management
461 
462 Rights can be assigned to the execution of actions in the user interface or to the editing
463 of domain classes and individual objects (instances of domain classes). For editing content the framework
464 defines the rights @em read, @em modify, @em delete and @em create.@n
465 The authorization for actions in the user interface is handled by the ActionMapper, for actions
466 concerning the data the PersistenceMapper checks the permissions (and sets objects, for which
467 the right @em modify is not set, to non-editable (@em is_editable = false) ).
468 Both classes use the method RightsManager::authorize and generate a fatal error message,
469 if authorization fails (see @ref howtoerror).
470 To prevent this the rights can be retrieved directly in order to take appropriate messures:
471 @verbatim
472  $rightsManager = &RightsManager::getInstance();
473  if ($rightsManager->authorize($this->_request->getValue('oid'), '', ACTION_READ))
474  {
475  $object = &$persistenceFacade->load($this->_request->getValue('oid'), BUILDDEPTH_INFINITE);
476  }
477  else
478  {
479  // do something else if the user cannot read the object
480  }
481 @endverbatim
482 
483 In the example the object is only loaded, if it's permitted. For the definition of rights
484 see @ref secauthorization.
485 @note If @em anonymous is set to one in the configuration file, the rights management
486 is disabled (see @ref seccms).

488 \subsection howtoconcurrency Concurrency
489 
490 If more than one user works with the application at the same time, conflicts can occur, if two
491 users want to edit the same object concurrently. In this case the first user can request
492 a lock on that object, which only allows reading access to succeeding users. This lock will
493 be transfered to all instances loaded in the future as long until the user unlocks the object,
494 which happens automatically upon execution of an action (call to main.php).
495 In the sourcecode this looks as follows:
496 @verbatim
497  $lockManager = &LockManager::getInstance();
498  $object = &$persistenceFacade->load($this->_request->getValue('oid'), BUILDDEPTH_INFINITE);
499 
500  // if the object is locked by another user we retrieve the lock to show a message
501  $lock = $object->getLock();
502  if ($lock != null)
503  {
504  $lockMsg .= $lockManager->getLockMessage($lock, $recipe->getName());
505  }
506  else
507  {
508  // try to lock object
509  $lockManager->aquireLock($this->_request->getValue('oid'));
510  }
511 @endverbatim
512 
513 The functionality described above is imlemented in the method LockManager::handleLocking,
514 with the only difference that this method only acquires a lock in cases in which the
515 user is allowed to modify the object.@n
516 An object, which is loaded by another user is set to non-editable (@em is_editable = false)
517 automatically upon loading.

519 \subsection howtoi18n Multilanguage Support
520 
521 For localization of the application the method Message::get is provided. For a given (english)
522 string this method searches for the required language's version. The language version can
523 either be passed directly with the method call or can be set application wide (see documentation
524 of Message::get).@n
525 How the translation is retrieved depends on the parameter @em usegettext (see @ref seccms).
526 If it is set to 1, the method uses the PHP function @em gettext. This in turn makes use of
527 *.mo files in the directory /localeDir/language/LC_MESSAGES - e.g. /locale/de_DE/LC_MESSAGES/main.mo
528 (see documentation of gettext).@n
529 If gettext doesn't exist, the language version can also be taken from an associative array named
530 messages_language (e.g. messages_de_DE).
531 This must be defined in a file /localeDir/language/LC_MESSAGES/messages_language.php (e.g.
532 /locale/de_DE/LC_MESSAGES/messages_de_DE.php). The keys of the array are the strings, which are
533 passed to the method Message::get, the values are the corresponding translations.@n
534 @note If a translation for a text doesn't exist, the string, which was passed to the method
535 Message::get, will be used.
536 
537 For making the localization more comfortable some tools are provided in the directory /wcmf/tools/i18n:
538 - @em locale.php extracts all strings to be localized (where the method Message::get is used) from the
539  application (views, controllers) and creates a *.po file for use with gettext.
540 - @em po2array.php generates the array definition described above from a *.po file.
541 
542 For localizing the view templates a Smarty plugin is provided, which is to be used as follows:
543 @verbatim {translate text="Logged in as %1%" r0=$authUser->getLogin()} @endverbatim
544

555 \subsection howtoerror Error handling
556 
557 In the application two types of errors are distinguished:
558 - @em Fatal: Errors, which are so critical, that it's not possible to proceed with the current action
559  (e.g. the missing of a controller).
560 - @em Non-fatal: Errors, which merely demand a notification to the user (e.g. invalid input)
561 
562 These errors can be produced in the following ways:
563
