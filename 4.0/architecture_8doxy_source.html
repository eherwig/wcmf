5 \section secarchitecture Architecture
6 
7 The main architectural concept of wCMF is the well known [Model-View-Controller]
8 (http://en.wikipedia.org/wiki/Model%E2%80%93view%E2%80%93controller) pattern.
9 The following sections describe it's implementation in wCMF and introduce some
10 other key concepts, that are useful to know when using wCMF.
11 
12 \subsection model Model
13 
14 An application usually consists of a _domain model_ that represents the real-world
15 concepts of the domain of interest. In object oriented programming this model is
16 implemented using classes. Depending on the application requirements the instances
17 of several of these classes have to be persisted in a storage in order to keep the
18 contained data.
19 
20 wCMF defines wcmf::lib::persistence::PersistentObject as base class for persistent
21 domain classes. It mainly implements an unique identifier for each instance (see \ref oid),
22 tracking of the persistent state, methods for setting and getting values as well as callback
23 methods for lifecycle events. For the composition of object graphs the wcmf::lib::model::Node
24 class is used as base class. It implements relation support for persistent objects.
25 
26 To retrieve persisted objects wcmf::lib::persistence::PersistenceFacade is used.
27 The actual operations for creating, reading, updating and deleting objects (e.g. SQL commands)
28 are defined in classes implementing the wcmf::lib::persistence::PersistenceMapper interface.
29 Although not necessary there commonly exists one mapper class for each persistent
30 domain class.

34 \subsubsection oid Object Identifier
35 
36 For handling lots of different domain objects an unique identification is crucial. In order to achieve this,
37 so-called _Object Identifiers_ (_OID_ or _object id_) are used. Various strategies are possible to obtain these
38 identifiers, for example a central registry. wCMF composes these object ids from the type (domain class or
39 type of the PersistentObject) and a number, which is unique for each type. Important is, that the object's
40 type can be derived from the object id, because this enables the PersistenceFacade to determine the
41 PersistenceMapper for the given object via the configuration. The class wcmf::lib::persistence::ObjectId
42 implements this concept.
43 
44 \subsubsection representations Model representations
45 
46 The output format of the nodes (e.g. when using the OutputVisitor) is specified in the corresponding
47 OutputStrategy. Examples are the XMLOutputStrategy for converting data to XML format for use in
48 @em Flash movies for instance or further processing via XSLT, TreeViewOutputStrategy for creating HTML
49 Treeviews or DotOutputStrategy for use with @em dot, which generates a graphic representation of the
50 data.

54 \subsection userinterface User interface
55 
56 The user interface of a wCMF application is based on the @em Model-View-Controller (@em MVC) pattern.
57 According to this pattern the application instanciates a specific controller for each action,
58 which loads the data (model) required for that action and - where applicable - presents the data
59 to the user via the corresponding view. Upon execution of a controller the next action will
60 be determined. This could either depend on the result of the action (@em successful, @em failed etc.)
61 or explicitly be specified by the user (@em save, @em delete etc.). The application flow is defined
62 by stringing together controller-action pairs.
63 
64 The ActionMapper - configured via a configuration file (see @ref configuration) - controls the application
65 flow. It creates @em controller objects, which execute actions like @em save changes, @em add
66 elements, @em delete elements or @em display data - where applicable - with the help of views. For
67 execution the Controller base class defines an algorithm, which is implemented by specialized
68 subclasses.@n
69 The @em model, which the controllers work with, is constituted of the domain classes - in most cases
70 a tree structure of Node objects.@n
71 The framework uses the <a href="http://smarty.php.net/" target="_blank">Smarty</a> Template Engine
72 to create the @em views. This means, that a view is defined by a @em Smarty template (*.tpl file),
73 which is displayed by an instance of the class View. Before the data is displayed the controller extracts
74 it from the model and assigns it to the view instance.

78 \subsection accesscontrol Access control
79 
80 Permissions for controller(|domain class|OID)-context-action triplets (see @ref configuration) are
81 configured based on roles (defined in the data storage). Several users can be assigned to one role
82 and each user can be assigned to different roles. The assignment of rights is made in the configuration file.
83 Here permissions can be explicitly granted or denied. During the application flow authentification/
84 authorization is accomplished by the RightsManager together with a representation of the current user (as an
85 instance of the AuthUser class, which is created upon successful login and added to the session data).
86 
87 \subsection applicationflow Application flow
88 
89 To run the wCMF in a web browser a PHP script is necessary as entry point. This is similar to a
90 @em main method, which instanciates the initial application objects and hands the control over
91 to these objects.
92 
93 In our case this script is @em main.php. It receives parameters (@em controller, @em context, @em action, @em data)
94 over a HTTP request. These parameters could for instance origin from the previous view. They are made available
95 to the %ActionMapper in form of an Request instance using the method ActionMapper::processAction. The %ActionMapper
96 then takes over the control of the application flow and returns after finishing its task or displaying a view, respectively.
97 Display of a view always interrupts the application's flow and allows the user to interact. The view sends the data obtained
98 during the interaction (e.g. form data) as HTTP Request back to the script @em main.php.@n
99 
100 For each call of the method ActionMapper::processAction the %ActionMapper on his part instanciates an appropriate
101 controller object, which is supplied with data by Controller::initialize and executed by Controller::execute.
102 The Controller baseclass provides in its method @em execute the algorithm for validating the data - and if
103 applicable - initializing the view. Concrete controller classes execute their specific tasks in the method
104 Controller::executeKernel.
105 If no view is assigned to a controller, the @em execute method returns @em true. The execution result is stored in the
106 Response instance, which again is converted into a Request instance and passed to the next call of the %ActionMapper's
107 @em processAction method. Thus it's possible to cascade different controllers to compose complex tasks out of several simple.
108 Otherwise the controller assigns the model to the view, which is then displayed by the view with its template (here @em view.tpl).
109 
110 Request and Response instances are interpreted using IFormat implementations. This allows the framework to realise
111 different formats for sending and receiving data. Known formats are HTML, JSON and SOAP.
112 
113 The configuration of the controllers, views and actions is done in the configuration file (see @ref configuration).
114
