5 \section secconfiguration Configuration
6 
7 The application is configured via one (or more) configuration file(s), which are parsed by the
8 WCMFInifileParser class. The configuration parameters are defined by key-value pairs as they are
9 used in the @em ini file format. The application interprets values, which are enclosed in curly brackets
10 (@em {}), as arrays. Related parameters are arranged in sections.
11 @note Please note, that the framework interprets paths - if not explicitly indicated otherwise - as relative
12 to the BASE path, which is defined in /application/base_dir.php. This is the root directory of /wcmf
13 and /application by default.

20 \subsection actionkey Action Keys
21 
22 An important concept of the application's configuration is that of @em Action @em Keys. An action key
23 describes the state of the application together with the action to be performed next. The state, which
24 the application is in, results from the current controller and the context, in which it is executed.
25 Controllers must exist as classes, whereas contexts and actions can be defined freely. They must only be
26 taken care of in the controllers and if necessary transported between the views (as form data, see also
27 @ref howtoapplication, @ref howtoviews).@n
28 The format of the action keys is the following:
29 
30 @verbatim controller?context?action @endverbatim
31 
32 In the configuration file an action key has the controller assigned, which should be executed next if
33 the action key is valid by the application's current state and the user action or the return value of
34 the current controller (see also @ref applicationflow).@n
35 Since parts of the action key can be omitted in the definition, an algorithm has to choose,
36 which action key fits a given value triplet. This algorithm is implemented in the method
37 WCMFInifileParser::getBestActionKey. In this method the %InifileParser searches in the configuration
38 file for the a list of combinations successively and ends the search when it found one. After
39 that the controller, which is assigned to that combination, is executed. The order of the combinations
40 is the following:
41 
42 @verbatim
43  1. controller?context?action
44  2. controller??action
45  3. controller?context?
46  4. ?context?action
47  5. ??action
48  6. controller??
49  7. ?context?
50 @endverbatim
51 
52 As a rule of thumb the action key which describes the state of the application the most
53 accurate is favoured.@n
54 Frequent examples are:@n@n
55 
56 Always execute the %SaveController when action @em save is selected, no matter which state the application is in:
57 @verbatim ??save = SaveController @endverbatim
58 
59 Always execute the %ProjektController when the context is @em project as long as no action is specified,
60 return to the ProjektController respectively from all controllers, which have no specific action assigned
61 upon termination:
62 @verbatim ?project? = ProjectController @endverbatim
63 
64 If no matching action key can be found, the %ActionMapper tries to execute the last %Controller that had a view attached again.
65 This can save some typing. For example you don't need to configure an action key for the @em ok return action of
66 SaveController, if you always want to execute the %Controller afterwards, from which the @em save action was initiated.

68 \subsection userconfig Configuration for individual users
69 
70 In the table @em user of the database scheme (see also @ref dbschema) each user can have her own configuration
71 file assigned. This will be loaded after the standard configuration (config.ini) and extends its settings.
72 If two keys have the same name the one from the user configuration overwrites the standard one. Using the @em roleconfig
73 configuration section, you can easily asign configuration files to groups (see below).

128 \subsubsection secinitparams [initparams]
129 Since some classes require specific parameters in their constructors (e.g. database connection),
130 these parameters can be grouped in sections (e.g. @em [database]) and referenced in the section
131 @em initparams. The keys are the classnames and the values the corresponding parameter sections.
132 The key-value pairs of the parameter section will be passed as associative array to the constructor
133 upon creation of the objects.
134 @verbatim class = initsection @endverbatim

140 \subsubsection secactionmapping [actionmapping]
141 In the section @em actionmapping the actual application flow is defined. This is achieved by defining
142 the transition from one controller to the next.@n
143 The key is an action key (see @ref actionkey), which clearly describes the state the application is
144 in before the transition. The value sets the next controller to execute. The ActionMapper uses this
145 information to determine which Controller is executed next.
146 @verbatim controllerA?context?action = controllerB @endverbatim

148 \subsubsection secviews [views]
149 For each controller a view can be defined dependent on its context and action, which is described
150 by its Smarty template (.tpl file). In the section @em views each action key can have a view template
151 assigned (see @ref actionkey). The controller uses this information to determine its view.
152 @verbatim controllerA?context?action = view @endverbatim

154 \subsubsection secauthorization [authorization]
155 The Authorization is based on action keys (see @ref actionkey), whereby the following actions can be
156 defined by the individual parts of the key:
157 @verbatim
158  controller?context?action ; an action in the application flow
159  type?context?action ; an action on a domain class (read, update, delete, create)
160  oid?context?action ; an action on a domain object (read, update, delete, create)
161 @endverbatim
162 To each of these actions different roles can be assigned (comma separated), for which a permission
163 is granted (@em +) or denied (@em -). The role @em * stands for all roles.@n
164 If a permission is not explicitly granted, it is considered as denied. For not listed Aktion Keys the
165 decision is made by the _defaultPolicy of the AuthUser. The RightsManager uses this information to
166 authorize an AuthUser for an action.
167 
168 The following code shows an example, where the creation of Category instances is only allowed for users having
169 the @em administrators role:
170 @verbatim Category??create = -authors +administrators @endverbatim
171 
172 \subsubsection secroleconfig [roleconfig]
173 Allows auto assignment of an additional configuration file for different user roles. If a user is added
174 to a listed role, then the given configuration file will be automatically assigned by the UserManager
175 (see also @ref userconfig).
176 @verbatim rolename = configurationfile @endverbatim
177 
178 For example if users of the @em administrators role should always use the admin.ini configuration file
179 you could add the following entry:
180 @verbatim administrators = admin.ini @endverbatim
