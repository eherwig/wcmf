5 # Model # {#model_main}
6 
7 wCMF applications are based on a model. The model defines the key aspects of the
8 application on a higher abstraction level than code does. You can think of it as
9 a condensed description of the application.
10 
11 A template based code generator transforms the model into source code that will
12 run inside the wCMF framework. Where necessary, the code can then be enhanced
13 manually inside pre-defined regions. Further generation runs will protect these
14 manual additions.
15 
16 We believe that this approach helps to develop a clear concept of the application
17 and improves code quality and maintainability dramatically.
18 
19 ## Modeling ## {#model_create}
20 
21 Technically the model is an [UML](http://en.wikipedia.org/wiki/Unified_Modeling_Language)
22 model that uses the _Chronos_ [profile](http://en.wikipedia.org/wiki/Profile_%28UML%29) from the
23 [Olympos](http://sourceforge.net/projects/olympos/) project. It is stored in
24 an XML file (e.g. `model.uml`). This file can be directly edited using
25 [Eclipse Papyrus](http://www.eclipse.org/papyrus/) or any other compatible UML
26 modeling tool. Alternatively you can use the
27 [Chronos Web Modeler](http://sourceforge.net/projects/olympos/) that allows
28 browser based collaborative modeling. wCMF uses a minimized version of the code
29 generator from the Olympos project with wCMF specific templates.
30 
31 The following diagram shows the workflow of the generator.
32 
33 \image html generator.png "Generator workflow"

35 ### Chronos profile ### {#model_profile}
36 
37 _Profiles_ are used to extend UML for different domains. They consist of __stereotypes__
38 and __tags__ that are applied to model elements in order to express domain specific
39 features. The following sections describe the stereotypes and tags of the _Chronos_
40 profile that are used to define various aspects of a wCMF application. The complete
41 [Chronos profile](https://raw.githubusercontent.com/iherwig/wcmf-default-app/master/model/chronos.profile.uml)
42 is available on GitHub.
43 
44 You will notice that only a small part of UML is used to define the application
45 and not all aspects of the application are defined in the model. Especially the
46 actual behavior of controllers and domain classes is omitted, since we believe
47 that this is more efficiently expressed in code. For simplicity and compatibility
48 reasons all aspects can be modeled in __class diagrams__ using __classes__,
49 __attributes__ and __associations__.
50 
51 \note UML elements and concepts that are not mentioned in the following sections
52 are most likely not supported by the generator and will be ignored.
53 
54 ### Domain classes ### {#model_classes}
55 
56 The following __stereotypes__ are used to model persistent domain classes and their relations.
57 
58 | Name | UML meta class | Description | Example
59 |------|----------------|-------------|--------
60 | _ChiNode_ | Class | Persistent domain class. Must inherit from the \link wcmf::lib::model::Node `Node`\endlink. | _Article_
61 | _ChiValue_ | Attribute | Persistent attribute in a _ChiNode_. | _headline_ of Article
62 | _ChiValueRef_ | Attribute | ReadOnly-reference to an attribute of another _ChiNode_. | _author_name_ in Article references _name_ in Author
63 | _ChiManyToMany_ | Class | Connection class in a many to many relation between _ChiNode_ classes. Must inherit from \link wcmf::lib::model::Node `Node`\endlink. | One _Author_ writes nultiple _Articles_ and one _Article_ has multiple _Authors_
64 | _ChiAssociation_ | Association | Optional, e.g. used to define a foreign key name in a relation between _ChiNode_ instances. | _author_id_ in Article as foreign key to Author table
65 
66 #### ChiNode / ChiManyToMany #### {#model_chinode}
67 
68 The following __tags__ are defined on the _ChiNode_ / _ChiManyToMany_ stereotypes.
69 
70 | Tag | Description | Example | Default value
71 |-----|-------------|---------|--------------
72 | _initparams_ | Name of the configuration section, which defines initialization parameters for the \link wcmf::lib::persistence::PersistenceMapper `PersistenceMapper`\endlink instance | _database_ | _database_
73 | _display_value_ | Attributes to display in a list view: A single attribute name or \|-separated list of attribute names | _name\|date_ | |
74 | _orderby_ | Definition of default sorting: _none_ (no order), _sortkey_ (generates a _sortkey_ column, that is used for explicit sorting) or the name of any attribute optionally followed by _ASC_ or _DESC_ | _name ASC_ | _none_
75 | _is_searchable_ | Boolean, indicating whether this type should be included in the default search | _true_ | _true_
76 | _is_soap_ | Boolean, indicating whether this type should be exposed to the SOAP interface | _true_ | _true_
77 | _table_name_ | The name of the database table in which instances will be stored | _Author_ | Class name
78 | _pk_name_ | The name of the primary key column: A single value or \|-separated list of values (the generator will add this automatically, if there is no appropriate attribute) | _fk_user_id\|fk_role_id_ | _id_
79 | _child_order_ | The order of the associated child types: A single type or \|-separated list of types | _Image\|Textblock\|Attachment_ in Article type | |
80 | _parent_order_ | The order of the associated parent types: A single type or \|-separated list of types | _Author\|Publisher_ in Article type | |
81 
82 \note Some of these tags are accessible in the application through the
83 \link wcmf::lib::persistence::PersistentObject::getProperty `PersistentObject::getProperty`\endlink
84 method.

86 #### ChiValue #### {#model_chivalue}
87 
88 The following __tags__ are defined on the _ChiValue_ stereotype.
89 
90 | Tag | Description | Example | Default value
91 |-----|-------------|---------|--------------
92 | _app_data_type_ | Application specific attribute tags: A single value or \|-separated list of values | _TAG_A\|TAG_B_ | _DATATYPE_ATTRIBUTE_
93 | _db_data_type_ | The attribute's database type | _TEXT_ | _VARCHAR(255)_ |
94 | _is_editable_ | Boolean, indicating whether this attribute is editable | _true_ | _true_
95 | _input_type_ | Name of the attribute's input control as listed in the _InputTypes_ configuration section | _filebrowser_ | _text_
96 | _display_type_ | Name of the attribute's display type as listed in the _DisplayTypes_ configuration section | _image_ | _text_
97 | _restrictions_match_ | Name of the attribute's validation type as listed in the _Validator_ configuration section | _regexp:^[0-9]*$_ | |
98 | _restrictions_not_match_ | _not used_ | | |
99 | _restrictions_description_ | Validation description used in case of a validation error | _The value must be an integer or empty_ | |
100 | _column_name_ | The name of the database column in which the attribute will be stored | _name_ | Attribute name
101 
102 \note Some of these tags are accessible in the application through the
103 \link wcmf::lib::persistence::PersistentObject::getValueProperty `PersistentObject::getValueProperty`\endlink
104 method.

116 
117 The following __tags__ are defined on the _ChiAssociation_ stereotype.
118 
119 | Tag | Description | Example | Default value
120 |-----|-------------|---------|--------------
121 | _fk_name_ | Name of the foreign key attribute (the generator will add this automatically, if there is no appropriate attribute) | _author_id_ | _fk_\_type\_ _id_ |

123 #### Example diagram ####
124 
125 A simple data model is shown in the diagram below. Each of the domain classes
126 _Author_, _Article_ and _Image_ is modeled with _ChiNode_ stereotype, their
127 attributes with _ChiValue_ stereotype. The only exception is the _author_name_ attribute,
128 which is a reference to the _name_ attribute of _Author_ and therefore uses the
129 stereotype _ChiValueRef_. As the relations between the classes show, one _Author_
130 can own several _Articles_ and each _Article_ in turn can contain several _Images_.
131 
132 All \link wcmf::lib::persistence::PersistenceMapper `PersistenceMapper`\endlink instance
133 are initialized using the parameters defined in the _database_ configuration section.
134 
135 In list views _Author_ instances are sorted by _name_ while _Article_ and _Image_
136 instances get an attribute _sortkey_ which is used to define an explicit order.
