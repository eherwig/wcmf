16 \subsection extpers Persistence mechanism
17 
18 Since the support of a persistence mechanism affects the loading, storing and deleting of
19 domain objects, corresponding PersistenceMapper subclasses have to be defined. The storing of
20 application data in a text file for instance requires a FileMapper for each domain class,
21 which is able to load, delete and alter objects in a text file.
22 
23 \subsection exttype Domain classes
24 
25 Adding new domain classes first of all requires their definition in the persistence mechanism
26 (name, attributes, contained domain classes). In a relational database this usually is achieved
27 by one table per domain class and if necessary additional connection tables, which reflect the
28 relations between the objects. In the next step a corresponding PersistenceMapper has to be implemented,
29 which makes the access to the data and the construction of the accompanying domain object possible.@n
30 If the Node class isn't sufficient for the data representation, a new domain class finally has to be
31 defined (for details see @ref howtotype).
32 
33 \subsection extoutput Output formats
34 
35 The output formats, in which the application data is transformed, can be defined via an OutputStrategy.
36 This strategy may be applied for example by using an OutputVisitor, to which the new format is made accessible
37 by its method OutputVisitor::setOutputStrategy.@n
38 If large amounts of data have to be exported, alternatively one of the subclasses of BatchController can be used,
39 since these allow the splitting of long tasks into packages, which can be handled over multiple page calls.
40 This is useful for example when exporting static HTML pages from a CMS.
41 
42 \subsection extinteraction User interaction
43 
44 For defining the user interaction in the simplest case solely one configuration file is needed, which
45 establishes an order of controllers for certain actions. By that a controller for a complex action
46 can be defined through the combination of primitive controllers (e.g. a MoveItemController from an
47 InsertItemController and a DeleteItemController).
48 If new actions should be added, a new controller has to be implemented (for details see @ref howtoapplication).@n
49 Another way to vary the user interaction is given through the modification of the views, which means the
50 modification of the form and link definitions in the templates. By defining different contexts it is also
51 possible to assign different views to a controller while maintaining the same action (e.g. to display an
52 input form for data in one context and the preview of the same data in another).
53 
54 \subsection extlayout Layout
55 
56 The application's appearance is completely described by the views. Since these themselves are defined in
57 corresponding Smarty templates, modifying the templates makes it quite easy to fit the layout to the special
58 needs of the application.
59 
60 \subsection extdataformat Data exchange formats
61 
62 Data sent to the framework and received from the framework is transported in Request and Response instances.
63 The data is interpreted using an IFormat implementation. This allows to easily implement new data exchange formats.
64 Already supported formats are HTML (HTMLFormat), JSON (JSONFormat) and SOAP (SOAPFormat).
65 */
