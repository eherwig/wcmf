5 # Persistence # {#pers_main}
6 
7 Persistence refers to that part of the domain model, which we could call the data model.
8 
9 ## Data model ## {#pers_datamodel}
10 
11 The data model consists of domain classes whose instances are saved in the storage.
12 The following steps need to be accomplished in order to make domain class instances
13 persistent.
14 
15 - Definition and creation of the database tables, if using a database storage.
16 - Implementation of the \link wcmf::lib::persistence::PersistenceMapper `PersistenceMapper`\endlink
17  classes, which map the domain classes to these tables, or anything else that is used as storage.
18 - Implementation of the domain classes as subclasses of \link wcmf::lib::model::Node `Node`\endlink.
19 - Configuration of the persistence layer.
20 
21 @note If you are using the generator to create the application from a model
22 (see \ref model), all necessary code will be generated automatically.
23 
24 ### Database tables ### {#pers_tables}
25 
26 Still the most common storage for web applications is a _SQL_ database (e.g.
27 [MySQL](http://www.mysql.com/)). It is also wCMF's default storage. Although not
28 strictly necessary it is recommended to use one database table for each persistent
29 class, where each class property maps to one table column and each row stores one
30 instance. The object identity is stored in the primary key column, which is named
31 _id_ by default.
32 
33 @note wCMF uses a table called `DBSequence` for retrieving the next id value used for
34 insertion, since _autoincrement_ columns are not supported on all database servers.
35 
36 #### Relations #### {#pers_relations}
37 
38 The following two relations types must be considered, when modeling the data
39 tables:
40 
41 - **One-To-Many** relations are realized by adding a foreign key column to the
42 child table (_many_-side), which points to the parent table (_one_-side). This
43 column is named _fk_ _ + parent table name + _ _id_ by default (e.g. _fk_author_id_).
44 
45 - **Many-To-Many** relations between two domain classes, are established by defining
46 a connection table. This table contains two primary keys, which point to the two
47 connected tables.
48 
49 ### Persistence Mappers ### {#pers_mappers}
50 
51 wCMF uses persistence mapper classes to communicate between the application and
52 the persistent storage (see \ref arch_main). These classes implement the
53 \link wcmf::lib::persistence::PersistenceMapper `PersistenceMapper`\endlink interface,
54 which defines methods for all persistence actions. By using this pattern wCMF
55 does not make any assumptions about the actual storage, which can be flat files,
56 a database or anything else a mapper is able to handle. This approach also makes
57 it easy to connect an existing storage to a newly created wCMF application.
58 
59 In order to simplify the implementation of mapper classes, wCMF already contains
60 a class hierarchy for a common mapping approach, which maps each concrete class
61 to one database table ([Concrete Table Inheritance]
62 (http://martinfowler.com/eaaCatalog/concreteTableInheritance.html)). In this
63 hierarchy \link wcmf::lib::model::mapper::RDBMapper `RDBMapper`\endlink handles
64 the communication with the relational database, while
65 \link wcmf::lib::model::mapper::NodeUnifiedRDBMapper `NodeUnifiedRDBMapper`\endlink
66 defines the actual mapping rules. Application developers simply need to implement
67 one subclass of \link wcmf::lib::model::mapper::NodeUnifiedRDBMapper `NodeUnifiedRDBMapper`\endlink
68 for each persistent domain class, which declares the mapping of the attributes and
69 relations of that class.
70 
71 @note All mapper classes, that are created by the generator are
72 \link wcmf::lib::model::mapper::NodeUnifiedRDBMapper `NodeUnifiedRDBMapper`\endlink
73 subclasses.
74 
75 ### Domain Classes ### {#pers_classes}
76 
77 Domain class instances hold the data that are used in the application and persisted
78 in the storage.
79 
80 Persistent domain classes either inherit from
81 - \link wcmf::lib::persistence::PersistentObject `PersistentObject`\endlink, which
82 is a container with value getter and setter methods, or from
83 - \link wcmf::lib::model::Node `Node`\endlink, which adds methods for managing
84 relations.
85 
86 These two classes are completely generic, the actual identity of the domain class -
87 that is the _properties_ and the _relations_ - are defined by the related
88 \link wcmf::lib::persistence::PersistenceMapper `PersistenceMapper`\endlink.
89 
90 So if no additional domain logic is required in the domain class, it would be
91 sufficient to use \link wcmf::lib::model::Node `Node`\endlink as domain class.
92 But in many cases you will want to execute custom code for \ref pers_hooks or
93 \ref pers_validation and therefor create a custom subclass of
94 \link wcmf::lib::model::Node `Node`\endlink.
95 
96 @note All domain classes, that are created by the generator are
97 \link wcmf::lib::model::Node `Node`\endlink subclasses.
98 
99 #### Persistence Hooks #### {#pers_hooks}
100 
101 Persistence hooks are methods that are called at certain points of the lifecycle of
102 an instance. The default implementation of these methods is empty - in fact their
103 sole purpose is to be overwritten in subclasses on order to implement special
104 functionality that should be executed at those points.
105 \link wcmf::lib::persistence::PersistentObject `PersistentObject`\endlink defines
106 the following persistence hooks:
107 
108 - `afterCreate`
109 - `beforeInsert`
110 - `afterInsert`
111 - `afterLoad`
112 - `beforeUpdate`
113 - `afterUpdate`
114 - `beforeDelete`
115 - `afterDelete`

121 ### Configuration ### {#pers_config}
122 
123 The configuration tells the application which mapper classes are responsible
124 for which domain classes. This assignment is defined in the `TypeMapping`
125 configuration section. The following example shows the appropriate entries
126 for the `Author` domain class:
127 
128 ~~~~~~~~~~~~~{.ini}
129 [TypeMapping]
130 app.src.model.Author = $app_src_model_AuthorRDBMapper
131 
132 [app_src_model_AuthorRDBMapper]
133 __class = app\src\model\AuthorRDBMapper
134 connectionParams = $database
135 
136 [Database]
137 dbType = sqlite
138 dbHostName = 127.0.0.1
139 dbName = app/test-db.sq3
140 dbUserName =
141 dbPassword =
142 dbCharSet = utf8
143 ~~~~~~~~~~~~~
144 
145 See \ref configuration for details about the configuration format.

147 ## Usage ## {#pers_node}
148 
149 The following sections show some basic examples for using the persistence layer.
150 
151 ### Loading objects ### {#pers_load}
152 
153 ~~~~~~~~~~~~~{.php}
154 $oid = new ObjectId('Author', 1);
155 
156 // load the Author instance with id 1
157 $author = ObjectFactory::getInstance('persistenceFacade')->load($oid);
158 
159 // PersistenceFacade will return null, if an instance does not exist
160 if ($author == null) {
161  echo("An Author with object id ".$oid." does not exist.";
162 }
163 ~~~~~~~~~~~~~
164 
165 In the example the _Author_ instance with id _1_ is loaded.
166 
167 #### Build depth #### {#pers_builddepth}
168 
169 When loading objects we generally distinguish between **eager** and **lazy** loading
170 (see [Lazy loading](http://en.wikipedia.org/wiki/Lazy_loading)).
171 By default \link wcmf::lib::persistence::PersistenceFacade `PersistenceFacade`\endlink
172 performs _lazy loading_ by using _virtual proxies_ (instances of
173 \link wcmf::lib::persistence::PersistentObjectProxy `PersistentObjectProxy`\endlink).
174 That means that related objects are retrieved from the store, only when they are
175 actually accessed. In order to perform _eager loading_, a
176 \link wcmf::lib::persistence::BuildDepth `BuildDepth`\endlink
177 value may be passed in the method calls:
178 
179 ~~~~~~~~~~~~~{.php}
180 $oid = new ObjectId('Author', 1);
181 
182 // load the Author instance with id 1 together with all related objects
183 $author = ObjectFactory::getInstance('persistenceFacade')->load($oid, BuildDepth::INFINITE);
184 ~~~~~~~~~~~~~
185 
186 In this example the _Author_ instance with id _1_ is loaded together with all related
187 objects recursively.
188 
189 Instead of a \link wcmf::lib::persistence::BuildDepth `BuildDepth`\endlink
190 value, an integer number may be passed indicating the depth of relations to load.
191 The following image illustrates the _build depth_ parameter for a simple model.
192 
193 \image html builddepth.png "Build depth"
194 
195 If using a build depth value of _1_ the _Author A_ instance will be loaded together
196 with it's related _Book_ instances (_Book A_, _Book B_). A value of _2_ will also load
197 the _Chapter_ instances (_Chapter A1_, _Chapter A2_). The default value is
198 \link wcmf::lib::persistence::BuildDepth::SINGLE `BuildDepth::SINGLE`\endlink,
199 which means that only the _Author A_ instance is loaded. In the above illustration
200 the value of _2_ is equal to passing
201 \link wcmf::lib::persistence::BuildDepth::INFINITE `BuildDepth::INFINITE`\endlink.
202 
203 ### Searching objects ### {#pers_search}
204 
205 To search objects in the store the
206 \link wcmf::lib::persistence::PersistenceFacade::loadObjects `PersistenceFacade::loadObjects`\endlink
207 method may be used. It allows to set the conditions that loaded objects should match.
208 
209 ~~~~~~~~~~~~~{.php}
210 $criteria = array(
211  new Criteria("Book", "title", "LIKE", "A%"),
212  new Criteria("Book", "year", ">=", "2014")))
213 )
214 
215 // load all Book instances with titles starting with A and release date 2014 or later
216 $books = ObjectFactory::getInstance('persistenceFacade')->loadObjects("Book", BuildDepth::SINGLE, $criteria);
217 ~~~~~~~~~~~~~
218 
219 In this example all _Book_ instances with titles starting with _A_ and release date
220 _2014 or later_ are loaded.
221 
222 For more complex cases the class \link wcmf::lib::model::ObjectQuery `ObjectQuery`\endlink
223 may be used for searching.
